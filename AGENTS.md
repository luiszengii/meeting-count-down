# AGENTS.md

## 项目名称

Feishu Meeting Countdown for macOS

## 项目概览

这是一个面向 macOS 的菜单栏常驻应用，用于读取飞书相关会议数据，在会前自动触发倒计时与音效提醒。当前产品定位不是“单一路径接飞书 API”，而是“多来源接入 + 本地统一规范化 + 本地提醒调度”：同一套客户端覆盖普通用户、管理员/开发者，以及完全拿不到开放权限的用户。

当前版本聚焦 macOS 与飞书会议场景，目标是先把“下一场会议提醒”做稳定，再扩展视觉和高级筛选能力。

## 仓库级持久规则

本文件是整个仓库的长期规则入口。未来任何新 session 进入本仓库后，都应先读根级 `AGENTS.md`，再读目标目录下的局部 `AGENTS.md`。如果局部目录规则与根规则冲突，以更靠近目标文件的 `AGENTS.md` 为准；若局部目录没有额外规则，则默认继承本文件。

### 文档系统总原则

1. 先建文档骨架，再开始大规模代码实现。
2. 任何新增代码目录，都必须先补该目录自己的 `AGENTS.md`，再往里面放核心代码。
3. 任何新增核心代码文件，都必须带中文模块级详细注释，目标是帮助后续开发者理解 Swift/macOS 生态、模块边界、状态流和 Apple 框架交互方式。
4. 任何会改变项目理解成本的工作，都必须同步更新 `docs/`，避免知识只留在聊天记录里。
5. 文档之间统一使用相对路径 Markdown 链接，逐步形成可浏览、可追踪的文档网络。

### 代码注释规则

注释风格统一采用“学习导向的模块级详细注释”，目标读者默认包含“完全不熟悉 Swift 语法和 Apple 框架的人”。注释不是为了装饰，而是为了降低未来阅读门槛。目标密度为：平均每 `8` 到 `10` 行有效代码，至少有 `1` 行真正提供信息量的注释；核心状态流、并发边界、框架交互点可以高于这个密度。

| 注释对象 | 要求 |
| --- | --- |
| 文件头 | 说明文件目的、所在模块、与其他模块的关系 |
| 重要类型 / 协议 | 说明职责、生命周期、状态含义、线程或 actor 约束 |
| 核心方法 | 默认必须写函数级注释，说明输入、输出、副作用、失败语义、为什么这样设计 |
| 关键计算属性 / 关键测试用例 | 默认必须写注释，说明为什么存在、对状态流或断言的意义 |
| Apple 框架交互点 | 说明框架语义、坑点、权限或生命周期前提 |
| 状态机 / 调度逻辑 | 说明状态转换、取消条件、重算时机 |

对于极短、语义完全直白的样板访问器，可以不逐个写函数级注释；但只要函数里出现条件分支、状态变更、异步边界、格式化逻辑、测试构造逻辑或 Apple 框架调用，就默认需要函数级注释。标准不是“作者自己看得懂”，而是“不会 Swift 的读者能靠注释快速跟上”。

## 当前结论

本项目当前采用“四路接入”架构，默认引导顺序为 `CalDAV -> BYO Feishu App -> Offline Import -> lark-cli`。

| 顺位 | 接入方式 | 目标用户 | 优点 | 主要限制 |
| --- | --- | --- | --- | --- |
| 1 | `CalDAV -> macOS Calendar` | 普通用户 | 不依赖开放平台建应用，最像普通用户可走通的路径 | 同步时机受系统日历控制，最后一分钟变更不如 API 直连可控 |
| 2 | `BYO Feishu App` 直连 | 管理员、IT、开发者 | 数据最全、可控性最高、适合准实时提醒 | 需要开放平台权限、配置应用、走 OAuth |
| 3 | `Offline Import` | 无开放权限用户 | 无审批依赖，落地最稳 | 不具备自动同步能力，需要重新导入 |
| 4 | `lark-cli` 辅助 | 技术用户、agent 工作流 | 适合诊断、导出、引导、开发调试 | 会受企业审批影响，不适合作为默认接入路径或运行时主链路 |

`lark-cli` 的角色已经明确收敛为“agent-friendly 辅助工具”，不内置到应用里，不作为长期后台轮询唯一依赖。

## 产品目标

用户价值是把“进入会议状态”前置：比飞书原生通知更强的会前提醒、更可定制的音效、更自动化的联动。产品目标是让用户在自己当前权限条件下，至少能走通一条会议接入路径，并稳定拿到“下一场会议”，再由本地客户端完成调度、播放与状态展示。

首版只聚焦三件事：能接入、能读会、能提醒。所有设计优先级服从这三点。

## 四路接入

### 1. CalDAV 模式

这是默认推荐路径。用户在飞书日历设置中生成 `Mac 专用密码`，把飞书日历作为 `Other CalDAV Account` 添加到 macOS Calendar，客户端通过 `EventKit` 读取系统日历中的飞书会议。服务器地址形态为 `caldav.feishu.cn`。这条路径适合普通用户，不要求飞书开放平台权限，但提醒准确性会受系统日历同步时机影响。

### 2. BYO Feishu App 模式

这是最完整、最可控的正式接入路径，适合管理员、IT、开发者或能自行创建飞书应用的高级用户。推荐应用类型为 `企业自建应用`；必需权限为 `calendar:calendar:readonly` 和 `offline_access`；必需安全配置为固定回调地址 `http://127.0.0.1:23388/oauth/callback`。客户端在本地完成 loopback OAuth、token 交换、refresh token 原子替换、Calendar API 同步和提醒调度。

### 3. Offline Import 模式

这是无权限环境下的兜底路径。用户手动导入 `.ics` 或后续支持的会议快照文件，客户端仅做本地解析、标准化、缓存和提醒，不承诺自动反映远端变更。任何会议调整都需要用户重新导入。

### 4. lark-cli 辅助模式

这是辅助路径，不是默认路径。客户端只检测并调用本机已安装的 `lark-cli`，不内置工具本身。它适合做权限探测、登录诊断、数据可达性验证、临时导出、agent 工作流接入和开发调试；不适合作为首版唯一后台常驻数据链路。

## 用户配置与首次启动

首次启动时客户端先进入接入向导。向导需要做四件事：选择接入方式、检测系统日历权限、检测本地回调端口、检测本机 `lark-cli` 是否存在。推荐顺序固定为：先尝试 `CalDAV`，若用户需要更高实时性或 `CalDAV` 不可用，再引导到 `BYO Feishu App`；若开放平台权限不可得，则退化到 `Offline Import`；`lark-cli` 只作为“辅助工具可用时补充显示”，不应被放在主引导路径前面。

| 模式 | 用户需要做的事 |
| --- | --- |
| `CalDAV` | 在飞书中生成 CalDAV 用户名、专用密码、服务器地址；在 macOS Calendar 添加账户；在客户端授予系统日历读取权限；选择目标系统日历 |
| `BYO Feishu App` | 创建企业自建应用；开通并发布 `calendar:calendar:readonly` 与 `offline_access`；配置回调地址；在客户端填入 `App ID` 和 `App Secret`；完成 OAuth 登录 |
| `Offline Import` | 获取 `.ics` 或会议快照文件；在客户端导入；后续按需重新导入 |
| `lark-cli` | 安装 `lark-cli`；登录并等待组织审批；允许客户端读取非敏感诊断结果或导入 CLI 导出的会议快照 |

## 统一行为规则

### 数据源与事件标准化

客户端在任一时刻只激活一个主数据源，不做多源混合调度。无论数据来自 API、系统日历、CLI 快照还是导入文件，都先进入统一的会议标准化层，再进入提醒引擎。用户切换接入方式时，必须立即取消旧调度并基于新数据源重建。

### 会议选择规则

默认取“当前时间之后最近的一场非全天会议”。首版以客户端规范化后的会议模型为准；循环日程按展开后的实例处理，不直接对抽象规则本体处理提醒。

### 倒计时与音效规则

倒计时秒数默认等于当前音效文件时长并向上取整；如果用户手动设置倒计时秒数，则以手动值为准；如果音效时长大于会议剩余时间，则立即触发播放，同时在 UI 中提示“音效长于剩余会议时间”；如果会议临近被修改或取消，则需要取消旧任务并重新计算。

### 刷新规则

启动应用后立即读取当前数据源，用户点击“立即刷新”后立刻重拉或重读。`BYO Feishu App` 模式默认 `120` 秒轮询，若下一场会议距离开始小于 `30` 分钟，则提升到 `30` 秒轮询，并在增量同步失败时回退到一次全量同步。`CalDAV` 模式依赖系统日历同步，同时结合 `EventKit` 变化通知与周期性回读。`Offline Import` 不自动刷新，只在重新导入时替换缓存。`lark-cli` 首版以手动触发和快照输入为主，不承诺长期后台轮询稳定性。睡眠唤醒、网络恢复、时区变化、手动修改配置后，都要立即重新计算。

## 技术栈与总体架构

### 技术栈

客户端语言为 Swift 5.10+ / Swift 6，UI 使用 SwiftUI，菜单栏以 `MenuBarExtra` 为主、必要时补 `NSStatusItem`；音频播放使用 AVFoundation / `AVAudioPlayer`；本地通知使用 UserNotifications；系统日历访问使用 EventKit；敏感信息进入 Keychain，非敏感偏好和缓存放在 UserDefaults + 文件系统；网络层使用 `URLSession`；本地 OAuth 回调监听使用固定端口 loopback HTTP 服务；如需调用外部 `lark-cli`，使用 `Process`。

### 总体架构

产品运行形态是“本地客户端 + 可插拔数据源”。macOS 客户端负责接入方式选择、`lark-cli` 探测与辅助调用、飞书应用配置、本地 OAuth 回调、token 管理、系统日历读取、离线导入解析、多源事件标准化、本地提醒调度、音效播放和系统集成。飞书开放平台负责用户授权、token 签发和 Calendar API；飞书日历客户端能力负责 CalDAV 账户同步；`lark-cli` 只负责辅助登录、权限诊断和数据可达性验证。

## 数据设计

敏感数据只进 Keychain。`BYO Feishu App` 模式需要保存 `app_id`、`app_secret`、`user_access_token`、`refresh_token`、`access_token_expires_at`、`refresh_token_expires_at`。`CalDAV` 模式优先由系统日历账户管理凭证，客户端不重复保存 CalDAV 密码。`lark-cli` 的登录态由 CLI 自己管理，客户端只读取非敏感诊断结果或导入文件。

本地非敏感数据至少包括：`active_connection_mode`、`selected_calendars`、`selected_system_calendar_ids`、`sync_token`、`last_successful_sync_at`、`last_cli_snapshot_at`、`recent_events_cache`、`imported_calendar_files`、`sound_profiles`、`reminder_preferences`、`window_state`。

本地模型统一为：`ConnectionMode`（`caldav_system_calendar`、`byo_feishu_app`、`offline_import`、`cli_assisted`）、`ConnectionProfile`、`FeishuAppConfig`、`AccountSession`、`SystemCalendarProfile`、`CLIProfile`、`ImportedCalendarProfile`、`SoundProfile`、`ReminderPreference`。数据设计重点不是字段数量，而是保证“任何来源的会议都能落到同一标准模型上”。

## 核心流程

### BYO Feishu App 授权与刷新

用户在客户端输入 `App ID` 与 `App Secret` 后，客户端启动本地 loopback 回调服务 `http://127.0.0.1:23388/oauth/callback`，构造飞书授权链接并打开浏览器。用户完成授权后，飞书把 `code` 重定向到本地回调；客户端解析 `code`，调用 `oauth/token` 获取 `user_access_token` 与 `refresh_token`，并写入 Keychain。后续在 access token 即将过期时，客户端调用 refresh token 接口获取新 token，并原子替换旧值。

### API 同步

客户端启动后先读取本地凭证与偏好设置；若 token 即将过期则先刷新；然后获取日历列表并选择目标日历，执行首次全量同步，保存 `sync_token`，后续在固定间隔内做增量同步。建议查询窗口覆盖当前时间前 `30` 分钟到当前时间后 `24` 小时，必要时扩展到 `48` 小时。

### CalDAV 同步桥接

用户先在飞书侧生成 CalDAV 专用配置，再在 macOS Calendar 中添加对应账户；客户端请求 EventKit 权限，识别同步后的系统日历，允许用户选择目标日历，再把系统日历会议实例标准化后送入提醒引擎。系统日历发生变化时，客户端需要重新计算下一场会议。

### Offline Import

用户选择 `.ics` 或支持的快照文件后，客户端解析源文件、转换为统一会议模型、更新本地缓存并重建提醒任务。重新导入时必须原子替换旧快照。

### lark-cli 辅助

客户端先检测本机是否存在 `lark-cli`；若存在，则显示“可用于诊断、导出、agent 辅助”的可选入口；若用户走 CLI 路线失败，则根据失败原因推荐 `CalDAV`、`BYO Feishu App` 或 `Offline Import`。CLI 成功并不意味着要把它升级为默认数据链路。

## 平台与分发

目标系统调整为 macOS 14 Sonoma 及以上，不再支持 macOS 13 Ventura；CPU 架构优先支持 Apple Silicon，建议提供 Universal Binary 兼容 Intel。开发阶段使用 Xcode 本地运行和 unsigned build；内测阶段提供签名后的 `.app` 或 `.dmg`；正式阶段要求 Developer ID 签名、Apple notarization、DMG 分发，自动更新建议用 Sparkle 2。

运行方式固定为：用户安装客户端后，首次启动看到四路接入选择；若用户选择 `CalDAV`，客户端读取系统日历；若选择 `BYO Feishu App`，客户端在本机完成 OAuth、refresh、同步和提醒；若选择 `Offline Import`，客户端读取本地文件并完成提醒；若用户本机已有 `lark-cli`，客户端把它作为辅助工具展示，而不是强制依赖。

## 工程原则

客户端本地闭环优先；运行时尽量减少外部强依赖；`lark-cli` 只做辅助不做主链路；每个用户的飞书应用配置互相隔离；敏感信息只进 Keychain 不进明文文件；首版先保证“下一场会议提醒”准确，再扩展视觉包装；先保证接入、同步、刷新稳定，再做更多产品层优化。

## 风险与注意事项

`CalDAV` 路线门槛最低，但存在系统同步延迟，最后一分钟变更不一定立即可见；`BYO Feishu App` 路线最完整，但要求用户具备开放平台配置能力；`Offline Import` 没有自动同步能力；`lark-cli` 可能卡在企业管理员审批，因此不应被设计成默认路径。除此之外，还必须重点防御以下问题：`offline_access` 缺失导致无法长期登录，refresh token 一次性使用导致刷新必须原子替换，本地 loopback 端口被占用，菜单栏应用在系统睡眠、唤醒后 timer 漂移，重复日程和例外实例处理错误，首次音频播放解码延迟，以及签名、notarization、自动更新这些分发硬依赖被拖到项目后期。

| 里程碑 | 判定标准 |
