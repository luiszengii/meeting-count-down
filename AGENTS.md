# AGENTS.md

## 项目名称

Feishu Meeting Countdown for macOS

## 项目概览

这是一个面向 macOS 的菜单栏常驻应用，用于读取飞书相关会议数据，在会前自动触发倒计时与音效提醒。当前产品定位不是“单一路径接飞书 API”，而是“多来源接入 + 本地统一规范化 + 本地提醒调度”：同一套客户端覆盖普通用户、管理员/开发者，以及完全拿不到开放权限的用户。

当前版本聚焦 macOS 与飞书会议场景，目标是先把“下一场会议提醒”做稳定，再扩展视觉和高级筛选能力。

## 仓库级持久规则

本文件是整个仓库的长期规则入口。未来任何新 session 进入本仓库后，都应先读根级 `AGENTS.md`，再读目标目录下的局部 `AGENTS.md`。如果局部目录规则与根规则冲突，以更靠近目标文件的 `AGENTS.md` 为准；若局部目录没有额外规则，则默认继承本文件。

### 文档系统总原则

1. 先建文档骨架，再开始大规模代码实现。
2. 任何新增代码目录，都必须先补该目录自己的 `AGENTS.md`，再往里面放核心代码。
3. 任何新增核心代码文件，都必须带中文模块级详细注释，目标是帮助后续开发者理解 Swift/macOS 生态、模块边界、状态流和 Apple 框架交互方式。
4. 任何会改变项目理解成本的工作，都必须同步更新 `docs/`，避免知识只留在聊天记录里。
5. 文档之间统一使用相对路径 Markdown 链接，逐步形成可浏览、可追踪的文档网络。

### 代码注释规则

注释风格统一采用“学习导向的模块级详细注释”，目标读者默认包含“完全不熟悉 Swift 语法和 Apple 框架的人”。注释不是为了装饰，而是为了降低未来阅读门槛。目标密度为：平均每 `8` 到 `10` 行有效代码，至少有 `1` 行真正提供信息量的注释；核心状态流、并发边界、框架交互点可以高于这个密度。

| 注释对象 | 要求 |
| --- | --- |
| 文件头 | 说明文件目的、所在模块、与其他模块的关系 |
| 重要类型 / 协议 | 说明职责、生命周期、状态含义、线程或 actor 约束 |
| 核心方法 | 默认必须写函数级注释，说明输入、输出、副作用、失败语义、为什么这样设计 |
| 关键计算属性 / 关键测试用例 | 默认必须写注释，说明为什么存在、对状态流或断言的意义 |
| Apple 框架交互点 | 说明框架语义、坑点、权限或生命周期前提 |
| 状态机 / 调度逻辑 | 说明状态转换、取消条件、重算时机 |

对于极短、语义完全直白的样板访问器，可以不逐个写函数级注释；但只要函数里出现条件分支、状态变更、异步边界、格式化逻辑、测试构造逻辑或 Apple 框架调用，就默认需要函数级注释。标准不是“作者自己看得懂”，而是“不会 Swift 的读者能靠注释快速跟上”。

## 当前结论

本项目当前采用“`CalDAV -> macOS Calendar -> EventKit`”单一路线。用户先在飞书里开启 CalDAV 同步，再把账户添加到 macOS 自带“日历”应用，客户端只读系统日历中的飞书会议并在本地完成提醒。

从现在开始，不再支持也不再继续扩展以下会议接入入口：

- `BYO Feishu App`
- `Offline Import`
- `lark-cli`

历史上曾讨论过多路接入，所以仓库里可能还残留 `ConnectionMode`、`SourceCoordinator`、多模式文案等历史骨架。从 `2026-03-31` 起，这些都视为待收敛的历史遗留，不再作为产品方向继续扩展。

## 产品目标

用户价值是把“进入会议状态”前置：比飞书原生通知更强的会前提醒、更可定制的音效、更自动化的联动。产品目标是让普通 macOS 用户只靠一条清晰、可执行的 CalDAV 路径就能稳定拿到“下一场会议”，再由本地客户端完成调度、播放与状态展示。

首版只聚焦三件事：能接入、能读会、能提醒。所有设计优先级服从这三点。

## 单一路径接入

### CalDAV 模式

这是默认推荐路径。用户在飞书日历设置中生成 `Mac 专用密码`，把飞书日历作为 `Other CalDAV Account` 添加到 macOS Calendar，客户端通过 `EventKit` 读取系统日历中的飞书会议。服务器地址形态为 `caldav.feishu.cn`。这条路径适合普通用户，不要求飞书开放平台权限，但提醒准确性会受系统日历同步时机影响。

用户侧的推荐操作流程应明确写给用户，而不是只写“支持 CalDAV”：

1. 在飞书里进入“设置 -> 日历 -> CalDAV 同步 -> 进入设置”。
2. 复制飞书提供的 `用户名`、`专用密码`、`服务器地址`。
3. 打开 macOS “日历”应用，进入“设置 -> 账户”。
4. 点击左下角 `+`，选择“其他 CalDAV 账户”。
5. 将账户类型设为“手动”。
6. 粘贴飞书提供的 `用户名`、`密码`、`服务器地址`，完成添加。
7. 成功后，账户列表里通常会出现 `caldav.feishu.cn` 供应商。
8. 用户可在 macOS 日历中把刷新间隔调到 `5 分钟` 或更短，以降低同步延迟。

产品和文档都应提前说明：这条路径的数据新鲜度受 macOS Calendar 同步节奏影响，最后几分钟内的会议变更不一定像 API 直连那样立刻可见。

## 用户配置与首次启动

首次启动时客户端不再让用户在多种接入方式之间做选择，而是直接进入 CalDAV 引导。向导需要做四件事：说明飞书侧和 macOS 侧的配置步骤、检测系统日历权限、枚举并识别已同步的系统日历、让用户选择目标飞书日历。

| 模式 | 用户需要做的事 |
| --- | --- |
| `CalDAV` | 在飞书“设置 -> 日历 -> CalDAV 同步”里复制用户名、专用密码、服务器地址；在 macOS 日历“设置 -> 账户”中通过“其他 CalDAV 账户 -> 手动”添加；确认出现 `caldav.feishu.cn` 供应商；按需把系统日历刷新间隔调到 `5` 分钟或更短；在客户端授予系统日历读取权限并选择目标系统日历 |

## V1 / V1.1 功能范围

### V1 必做

V1 的核心是“单一路径接入 + 单一下一场会议 + 稳定提醒”。具体包括：首次启动 CalDAV 引导；系统日历权限申请；目标飞书日历选择；系统日历读取；菜单栏状态展示；下一场会议计算；手动刷新；定时重读；睡眠唤醒后重算；系统日历变化后重算；音效导入、播放测试、静音模式、总开关、开机启动，以及点击提醒后打开会议链接或对应事件详情。

### V1.1 建议

V1.1 再做多套音效方案、指定多个系统日历生效、跳过全天事件、跳过已取消事件、仅对有视频会议信息的事件生效、仅对“已接受”或“未拒绝”的会议生效、会议前系统通知、会议开始后自动停止倒计时态、同步新鲜度提示和更完整的健康状态页。

## 统一行为规则

### 数据源与事件标准化

客户端运行时只读取系统日历中的飞书会议，不再支持多源混合调度，也不再提供数据源切换。所有会议都先从 `EventKit` 事件转换成统一的 `MeetingRecord`，再进入提醒引擎。

### 会议选择规则

默认取“当前时间之后最近的一场非全天会议”。首版以客户端规范化后的会议模型为准；循环日程按展开后的实例处理，不直接对抽象规则本体处理提醒。

### 倒计时与音效规则

倒计时秒数默认等于当前音效文件时长并向上取整；如果用户手动设置倒计时秒数，则以手动值为准；如果音效时长大于会议剩余时间，则立即触发播放，同时在 UI 中提示“音效长于剩余会议时间”；如果会议临近被修改或取消，则需要取消旧任务并重新计算。

### 刷新规则

启动应用后立即读取系统日历，用户点击“立即刷新”后立刻重读。默认空闲状态下每 `120` 秒重读一次本地系统日历；若下一场会议距离开始小于 `30` 分钟，则提升到每 `30` 秒重读一次。`CalDAV` 远端同步由 macOS Calendar 负责，本 app 只能尽量更频繁地重读本地系统日历，不能强制飞书立刻下发最新数据。系统日历事件变化、睡眠唤醒、网络恢复、时区变化、手动修改目标日历后，都要立即重新计算。

## 技术栈与总体架构

### 技术栈

客户端语言为 Swift 5.10+ / Swift 6，UI 使用 SwiftUI，菜单栏以 `MenuBarExtra` 为主、必要时补 `NSStatusItem`；音频播放使用 AVFoundation / `AVAudioPlayer`；本地通知使用 UserNotifications；系统日历访问使用 EventKit；非敏感偏好和缓存放在 UserDefaults + 文件系统；启动项使用 ServiceManagement。

### 总体架构

产品运行形态是“飞书 CalDAV 账户 -> macOS Calendar -> 本地菜单栏客户端”。macOS Calendar 负责账户保存和远端同步，客户端负责系统日历权限、目标日历选择、会议标准化、本地提醒调度、音效播放和系统集成。客户端不保存飞书开放平台凭证，也不引入自建应用 OAuth 或 CLI 运行时依赖。

### 模块划分

## 数据设计

敏感数据不进入本 app。CalDAV 用户名、专用密码和账户同步都由 macOS Calendar 自己管理，客户端不重复保存飞书密码，也不保存任何飞书 OAuth token。

本地非敏感数据至少包括：`selected_system_calendar_ids`、`last_successful_refresh_at`、`recent_events_cache`、`sound_profiles`、`reminder_preferences`、`window_state`、`onboarding_state`。

本地模型统一围绕 `SystemCalendarProfile`、`MeetingRecord`、`SoundProfile`、`ReminderPreference` 展开。历史上如果代码里还保留多源模型，新工作应优先收敛这些命名和状态，而不是继续扩展它们。

## 核心流程

### CalDAV 同步桥接

用户先在飞书侧生成 CalDAV 专用配置，再在 macOS Calendar 中添加对应账户；客户端请求 EventKit 权限，识别同步后的系统日历，允许用户选择目标日历，再把系统日历会议实例标准化后送入提醒引擎。系统日历发生变化时，客户端需要重新计算下一场会议。

面向用户的文档和向导文案应尽量贴近真实 macOS 操作路径：`日历 -> 设置 -> 账户 -> 左下角 + -> 其他 CalDAV 账户 -> 手动`。不要只写成抽象的“请添加 CalDAV 账户”，否则普通用户很难自行走通。

## 平台与分发

目标系统调整为 macOS 14 Sonoma 及以上，不再支持 macOS 13 Ventura；CPU 架构优先支持 Apple Silicon，建议提供 Universal Binary 兼容 Intel。开发阶段使用 Xcode 本地运行和 unsigned build；内测阶段提供签名后的 `.app` 或 `.dmg`；正式阶段要求 Developer ID 签名、Apple notarization、DMG 分发，自动更新建议用 Sparkle 2。

运行方式固定为：用户安装客户端后，先在飞书和 macOS 日历中完成 CalDAV 配置，再打开客户端授予系统日历权限、选择目标日历并开始提醒。

## 工程原则

客户端本地闭环优先；系统原生集成优先；不引入自建应用 OAuth、不引入 CLI 运行时依赖、不引入离线导入兜底；飞书 CalDAV 凭证只由 macOS Calendar 管理；首版先保证“下一场会议提醒”准确，再扩展视觉包装；先保证接入、同步、刷新稳定，再做更多产品层优化。

## 风险与注意事项

`CalDAV` 路线门槛最低，但存在系统同步延迟，最后一分钟变更不一定立即可见；用户可能只在飞书里开了 CalDAV 却没有在 macOS Calendar 中真正添加账户；系统里可能出现多个相似日历导致用户选错；菜单栏应用在系统睡眠、唤醒后容易出现 timer 漂移；重复日程和例外实例仍是提醒逻辑最容易出错的部分；首次音频播放存在解码延迟；签名、notarization、自动更新仍然是面向他人分发时的硬依赖。
