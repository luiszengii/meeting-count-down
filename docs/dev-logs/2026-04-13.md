## 详细内容

### 1. 完成事项

- 更新 [`scripts/configure-release-secrets.sh`](../../scripts/configure-release-secrets.sh)，不再写死 `login.keychain-db`，而是自动从默认 keychain、login keychain 和 `~/Library/Keychains/` 下的候选文件里寻找真正包含目标代码签名 identity 的 keychain。
- 修复本机用户级 keychain 搜索列表与默认 keychain：此前临时验证流程把 `security default-keychain` 和 `list-keychains` 指向了一个已经不存在的临时 keychain，导致后续 `security` 命令和 secrets 配置脚本都找不到 `Lingjun Zeng` identity。
- 在确认 `gh auth login` 已恢复后，实际执行 [`scripts/configure-release-secrets.sh`](../../scripts/configure-release-secrets.sh) 并成功写入以下 GitHub Repo Secrets：
  - `MACOS_SIGNING_IDENTITY`
  - `MACOS_CERTIFICATE_PASSWORD`
  - `MACOS_KEYCHAIN_PASSWORD`
  - `MACOS_CERTIFICATE_P12_BASE64`

### 2. 关键决策

- `configure-release-secrets.sh` 的默认行为改成“按 identity 反查可用 keychain”，而不是要求维护者自己知道登录钥匙串的精确文件名；因为本机环境里已经出现了 `login_renamed_1.keychain-db` 这类非默认命名。
- 继续保留脚本的 `--keychain-path` 参数，允许维护者在极端场景下显式指定来源 keychain，但常规路径优先自动探测。

### 3. 遇到的问题

- 这次暴露出的真正问题不是 `gh secret set`，而是本机 keychain 状态被此前调试命令污染：默认 keychain 指向临时路径后，`security login-keychain` 和脚本里写死的 `login.keychain-db` 都会给出误导性错误。
- 另外，`security export` 是否能顺利导出 `.p12` 仍受本机钥匙串 ACL 约束；脚本已经把这件事明确成“可能弹出系统授权框，需要维护者点允许”的前提。
