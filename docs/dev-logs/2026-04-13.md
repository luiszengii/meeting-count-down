## 详细内容

### 1. 完成事项

- 更新 [`scripts/configure-release-secrets.sh`](../../scripts/configure-release-secrets.sh)，不再写死 `login.keychain-db`，而是自动从默认 keychain、login keychain 和 `~/Library/Keychains/` 下的候选文件里寻找真正包含目标代码签名 identity 的 keychain。
- 修复本机用户级 keychain 搜索列表与默认 keychain：此前临时验证流程把 `security default-keychain` 和 `list-keychains` 指向了一个已经不存在的临时 keychain，导致后续 `security` 命令和 secrets 配置脚本都找不到 `Lingjun Zeng` identity。
- 在确认 `gh auth login` 已恢复后，实际执行 [`scripts/configure-release-secrets.sh`](../../scripts/configure-release-secrets.sh) 并成功写入以下 GitHub Repo Secrets：
  - `MACOS_SIGNING_IDENTITY`
  - `MACOS_CERTIFICATE_PASSWORD`
  - `MACOS_KEYCHAIN_PASSWORD`
  - `MACOS_CERTIFICATE_P12_BASE64`

### 2. 关键决策

- `configure-release-secrets.sh` 的默认行为改成“按 identity 反查可用 keychain”，而不是要求维护者自己知道登录钥匙串的精确文件名；因为本机环境里已经出现了 `login_renamed_1.keychain-db` 这类非默认命名。
- 继续保留脚本的 `--keychain-path` 参数，允许维护者在极端场景下显式指定来源 keychain，但常规路径优先自动探测。

### 3. 遇到的问题

- 这次暴露出的真正问题不是 `gh secret set`，而是本机 keychain 状态被此前调试命令污染：默认 keychain 指向临时路径后，`security login-keychain` 和脚本里写死的 `login.keychain-db` 都会给出误导性错误。
- 另外，`security export` 是否能顺利导出 `.p12` 仍受本机钥匙串 ACL 约束；脚本已经把这件事明确成“可能弹出系统授权框，需要维护者点允许”的前提。

### 4. 后续动作

## 补充记录（Release 构建修复）

- 更新 [LaunchAtLoginController.swift](../../MeetingCountdownApp/AppShell/LaunchAtLoginController.swift) 和 [SystemCalendarAccess.swift](../../MeetingCountdownApp/SystemCalendarBridge/SystemCalendarAccess.swift)，把长期只在主线程 UI / EventKit 桥接里使用的 `SMAppService`、`EKEventStore` 存储属性标成 `nonisolated(unsafe)`，消除 GitHub Actions `Release` 构建在 Swift 6 下对异步系统 API 的“跨 actor 发送”误判。
- 更新 [ReminderAudioEngine.swift](../../MeetingCountdownApp/ReminderEngine/ReminderAudioEngine.swift)，把 `AVAudioPlayerNode.scheduleBuffer` 调整回同步 completion-handler 版本，避免 `Release` 构建把 `AVAudioPCMBuffer` 和 `AVAudioPlayerNode` 误判成跨 actor 发送。
- 在推送 `v0.1.0` 后拉取 Actions 失败日志，确认首轮发布失败根因不是 secrets 或 `gh release`，而是三处仅在 `Release` / CI 环境暴露的 Swift 6 并发编译错误。

### 6. 关键决策

- 这次修复优先采用“最小范围解除错误发送检查”的做法，而不是立刻重写 `LaunchAtLoginController`、`SystemCalendarAccess` 或 `ReminderAudioEngine` 的 actor 边界；因为这些对象在当前架构里仍然只通过主线程入口被串行使用，问题主要是新 SDK 对系统框架类型的静态分析更严格。
- `ReminderAudioEngine` 继续保留 `@MainActor` 封装，不把音频调度改成新 actor 或后台队列；当前目标只是让 CI `Release` build 和本地运行时行为重新对齐。

## 补充记录（Debug / Release 权限主体隔离）

### 11. 完成事项

- 根据真实用户反馈和 `/Applications/FeishuMeetingCountdown.app` 的签名检查，确认当前 GitHub Release `v0.1.0` 已经是 signed 产物，但运行时日志仍稳定报 `尚未决定是否允许访问日历`，说明问题已经从“unsigned DMG 承接不了权限”收敛成“当前运行中的 release app 没有命中正确的 TCC 授权主体”。
- 更新 [`project.yml`](../../project.yml)，把 `MARKETING_VERSION` 提升到 `0.1.1`，同时把 Debug 配置切到独立 bundle identifier：`com.luiszeng.meetingcountdown.dev`，并把 Debug 显示名改成 `Feishu Meeting Countdown Dev`。
- 新增 [`shared-bundle-id-between-debug-and-release-confuses-calendar-permission.md`](../pitfalls/shared-bundle-id-between-debug-and-release-confuses-calendar-permission.md)，把“开发版和 release 版共用 bundle id 会让 Calendar / EventKit 权限排查变形”沉淀成正式 pitfall，并从 [pitfalls/README.md](../pitfalls/README.md) 和 [index.md](../index.md) 接入。

### 12. 关键决策

- 从这次 `v0.1.1` 开始，开发版和 release 版不再共用同一个 bundle id；否则只要两边签名主体不同，macOS TCC 就可能把“系统设置里看起来已授权”和“当前运行中的 app 真实拿到的授权状态”分裂开。
- 以后 release 渠道继续保持 `com.luiszeng.meetingcountdown` 稳定不变，开发调试链路一律使用 `.dev` 后缀的独立 app 身份。

### 13. 遇到的问题

- 这次比 unsigned DMG 更隐蔽：因为 release 包本身已经是 signed 的，所以光看“签没签名”已经不足以解释权限问题；必须进一步看到“本地开发版”和“release 版”其实还在抢同一条 bundle id。
- 用户侧系统设置截图只能说明“有一个同名 app 看起来被授权了”，不能直接证明当前运行中的 release app 真命中了同一条 TCC 记录。

### 14. 后续动作

- 基于这个修复发 `v0.1.1`，让 GitHub Release 用户安装版和本地调试版彻底分离权限主体，再验证 release 包里的 Calendar / EventKit 表现。
- 如果 `v0.1.1` 后仍有个别机器复现，就继续让用户从新的“复制诊断信息”入口反馈 bundle id、授权状态和可见日历列表，避免再次退回“只看截图猜原因”。
