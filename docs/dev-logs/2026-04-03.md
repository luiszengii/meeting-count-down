### 2. 关键决策

- 当前 DMG 只追求“比裸 app / zip 更像 macOS 安装包”的包装形态，不追求背景图、自定义窗口布局或伪装成正式签名版本；这样可以先提升测试用户的感知质量，同时继续保持脚本稳定和低维护成本。
- 图标先补一版能工作的占位方案，而不是继续空着：当前选择 `Lucide` 的 `calendar-clock`，是因为它同时覆盖“日历 + 时间”语义，比单纯铃声或时钟更贴近这个产品；等将来有正式视觉稿时，再替换掉这套占位资源。
- `Phase 6` 里的 unsigned `.app` / `.zip` / `.dmg` 只能继续承担“安装包装和首次放行”验证，不再把它们当成跨设备系统权限测试包；只要测试目标包含另一台机器上的 Calendar / EventKit 访问，就必须先给导出包补稳定签名身份。

### 3. 遇到的问题

- `DMG` 能改善包装形态，但不改变未签名、未 notarize 这个事实；因此文档里必须继续强调首次打开放行步骤，避免测试用户误以为“有了 DMG 就等于已经被系统信任”。
- 另一台机器上“系统设置里已经打开 Calendar 权限，但 app 仍然读不到日历”的现象，最终不是业务代码 bug，而是 unsigned 分发包没有稳定代码身份，不能可靠承接跨设备 TCC 权限。
- 在真正看到运行时截图后确认，当前 UI 的主要问题不是功能缺口，而是“状态层级太平、帮助文字太多、主操作不够明确”；因此第一轮优化先不追求视觉装饰，而是优先重构信息架构和主次操作关系。
