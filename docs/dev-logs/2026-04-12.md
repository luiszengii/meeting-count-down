# 2026-04-12

## 当天开发概括

把当前 `Phase 6` 的“手动分发准备”继续推进到 GitHub 平台侧：新增本地 `gh` 发布脚本、tag 驱动的 GitHub Actions Release workflow，并把签名 secrets 接入路径和维护者文档补齐。

## 详细内容

### 1. 完成事项

- 新增 [`.github/AGENTS.md`](../../.github/AGENTS.md) 和 [`.github/workflows/AGENTS.md`](../../.github/workflows/AGENTS.md)，为新的 GitHub 平台配置目录和 workflow 目录补齐职责边界、依赖和开发注意事项。
- 新增 [`.github/workflows/release.yml`](../../.github/workflows/release.yml)，让仓库在推送 `vX.Y.Z` tag 时自动校验签名 secrets、重新生成 Xcode 工程、导入 `.p12` 证书、调用仓库脚本构建 signed 产物，并创建或更新同名 GitHub Release。
- 新增 [`scripts/release-gh.sh`](../../scripts/release-gh.sh)，把本地 / CI 共用的 release 行为收口到一个脚本里：统一校验 git 工作区、tag 格式、`project.yml` 版本一致性，并通过 `gh release create` / `gh release upload --clobber` 处理 release 资产上传。
- 新增 [`scripts/configure-release-secrets.sh`](../../scripts/configure-release-secrets.sh)，把“从本机 keychain 导出现有 identity、生成密码并写入四个 GitHub secrets”收口成一个维护者脚本，避免每次都手拼 `security export` 和 `gh secret set`。
- 更新 [scripts/AGENTS.md](../../scripts/AGENTS.md)、[README.md](../../README.md)、[AGENTS.md](../../AGENTS.md)、[docs/README.md](../README.md)、[docs/index.md](../index.md) 和 [manual-installation.md](../manual-installation.md)，把“GitHub Release + tag 自动发布 workflow”纳入当前 `Phase 6` 的正式文档描述，并补上维护者入口。

### 2. 关键决策

- Release automation 继续复用现有 [export-release.sh](../../scripts/export-release.sh) 和 [create-dmg.sh](../../scripts/create-dmg.sh) 的打包逻辑，而不是在 GitHub Actions YAML 里重复维护一套构建 / 命名规则。
- 版本号以 git tag 为准，但 workflow 仍强制校验 [project.yml](../../project.yml) 的 `MARKETING_VERSION` 与 tag 去掉前缀 `v` 后完全一致；这样可以避免“GitHub Release 版本”和 app 内部版本漂移。
- 当前自动发布默认要求仓库已配置签名相关 GitHub secrets，不再把纯 unsigned 产物当成面向别人的默认 release 形态；这与 [unsigned DMG 在另一台 Mac 上无法稳定承接 Calendar 权限](../pitfalls/unsigned-dmg-calendar-permission-on-other-mac.md) 的结论保持一致。

### 3. 遇到的问题

- 仓库此前没有 `.github/workflows/`，所以这次不仅要补 workflow 本身，还要先补新目录的 `AGENTS.md`，避免平台自动化变成“无上下文的新代码孤岛”。
- Release 自动化一旦引入签名，就必须同时考虑本地脚本和 CI 的行为一致性；如果 CI 直接手写 `codesign + gh release` 而本地仍走另一套脚本，后续很容易出现资产命名和版本校验分叉。

### 4. 后续动作

- 下一步需要把当前本机可用的自签名 identity 导出为 `.p12`，并把 `MACOS_CERTIFICATE_P12_BASE64`、`MACOS_CERTIFICATE_PASSWORD`、`MACOS_SIGNING_IDENTITY`、`MACOS_KEYCHAIN_PASSWORD` 写入 GitHub Repo Secrets。
- 完成 secrets 配置后，推一个测试 tag 验证整条链路：workflow 成功构建、创建 release、上传 signed `.zip` / `.dmg`，并在另一台 Mac 上继续确认 Calendar / EventKit 权限表现。
