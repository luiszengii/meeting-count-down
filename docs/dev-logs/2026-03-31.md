# 2026-03-31

## 当天开发概括

今天把项目产品范围正式收敛成 `CalDAV-only`，不再保留 `BYO Feishu App`、`Offline Import`、`lark-cli` 作为会议接入入口。根级文档已经全部改写到单一路线。

## 详细内容

### 1. 根文档收敛为单一路线

- 重写了 [根级 AGENTS](../../AGENTS.md) 中从“当前结论”开始的产品、架构、开发计划和测试计划描述。
- 删除了根文档里关于 `BYO Feishu App`、`Offline Import`、`lark-cli` 作为会议接入入口的产品级描述。
- 保留并强化了 `CalDAV -> macOS Calendar` 的真实用户操作步骤。

### 2. README 改为用户视角的 CalDAV-only 说明

- 重写了 [README.md](../../README.md)，只保留 CalDAV 路线。
- README 现在明确告诉用户：app 不管理飞书 OAuth，不保存飞书密码，只读取已经同步到 macOS Calendar 的会议。

### 3. 新增产品范围收敛 ADR

- 新增 ADR：[2026-03-31 CalDAV-only 产品范围收敛](../adrs/2026-03-31-caldav-only-product-scope.md)。
- 这条 ADR 用来覆盖此前关于多路接入的产品级规划，并指导后续代码和测试继续收敛。

### 4. 下一步

- 代码层还残留多路接入历史骨架，后续应继续把状态模型、设置页和测试计划向单一路线收敛。
- 在真正开始功能开发前，优先保证 `EventKit` 读取、目标日历选择和下一场会议计算这条链路清晰稳定。

### 5. 完成 CalDAV-only 代码清理

- 删除了 `ConnectionMode`、`OnboardingRouter`、`OAuthLoopbackConfiguration` 以及对应测试，代码层不再保留 `BYO Feishu App`、`Offline Import`、`lark-cli` 的入口骨架。
- 把 [SourceCoordinator](../../MeetingCountdownApp/SourceCoordinator/SourceCoordinator.swift) 收敛成单一 `MeetingSource` 协调器，把 [SettingsView](../../MeetingCountdownApp/AppShell/SettingsView.swift) 改成只围绕 CalDAV 配置、系统日历权限和目标日历选择。
- 把 [PreferencesStore](../../MeetingCountdownApp/Preferences/PreferencesStore.swift) 收敛成只保存提醒偏好和系统日历选择；把 [Diagnostics](../../MeetingCountdownApp/Diagnostics/DiagnosticCheckers.swift) 收敛成只保留系统日历权限检查。
- 同步更新了 [MeetingCountdownApp](../../MeetingCountdownApp/AGENTS.md) 及多个子目录的 `AGENTS.md`，避免仓库继续把多入口历史骨架当成未来方向。

### 6. 修复日历权限使用说明键

- 在 [project.yml](../../project.yml) 里补上 `NSCalendarsFullAccessUsageDescription`，让 macOS 能正确展示“授权访问日历”所需的系统权限说明。
- 这不是新的功能范围，而是把已经接上的 `EventKit` 权限申请链路补全为系统可用状态。

### 7. 收紧默认推荐日历规则

- 把 [SystemCalendarAccess.swift](../../MeetingCountdownApp/SystemCalendarBridge/SystemCalendarAccess.swift) 里的默认推荐逻辑收紧为只认 `caldav.feishu.cn` 来源。
- 这样设置页只会把真正来自飞书 CalDAV 账户的系统日历标成“推荐”，不会把用户其它 `CalDAV` 或 `iCloud` 日历误标成推荐。

### 8. 修正设置窗口打开方式

- 菜单栏里的“打开设置”之前为了把窗口置前，错误地调用了 AppKit 的 `showSettingsWindow:`，运行时会报出“Please use SettingsLink for opening the Settings scene.”。
- 现已改成只走 SwiftUI 官方支持的 `openSettings()`，并新增 [SettingsWindowController.swift](../../MeetingCountdownApp/AppShell/SettingsWindowController.swift) 记录真实设置窗口，再由设置页自身把该窗口提到前台。
- 这样既避免了系统告警，也保留了菜单栏 `LSUIElement` 应用里“设置窗口不要落到其它窗口后面”的修复目标。


### 9. 补充本轮 session 的 Pitfalls 文档

- 新增 [EventKit 日历权限调试边界](../pitfalls/eventkit-calendar-permission-debugging.md)，记录“权限是按 app 单独授权”以及 `requestFullAccessToEvents()` 需要 `NSCalendarsFullAccessUsageDescription` 这两个容易混在一起的点。
- 新增 [SwiftUI Settings scene 在菜单栏 app 里的打开方式](../pitfalls/swiftui-settings-scene-in-menu-bar-app.md)，避免后续再回退到 `showSettingsWindow:` 这条不受支持的路径。
- 新增 [新增源文件后需要重新生成 Xcode 工程](../pitfalls/xcodegen-regenerate-project-after-adding-files.md)，记录本仓库使用 `project.yml -> xcodegen generate -> .xcodeproj` 的工程更新步骤。
