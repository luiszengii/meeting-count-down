## 补充记录（菜单栏图标与分发包体积）

### 5. 完成事项

- 更新 [MenuBarStatusItemController.swift](../../MeetingCountdownApp/AppShell/MenuBarStatusItemController.swift)，把菜单栏普通态和会前预热态的文字 / SF Symbol 前景色交回系统模板渲染；只在红色提醒胶囊态显式指定白色前景，避免状态栏图标长期被锁成黑色，并让副屏菜单栏的灰态表现重新跟随系统。
- 更新 [export-release.sh](../../scripts/export-release.sh) 和 [create-dmg.sh](../../scripts/create-dmg.sh)，在导出分发包时统一清理 app bundle 上的 FinderInfo / resource fork / provenance 等本机元数据，并改用 `ditto --norsrc` 复制 app，降低本地签名包在打 zip / DMG 时再次被扩展属性污染的概率。
- 同步修正分发脚本里 `signed / unsigned` 的包名判定：不再把“strict 校验是否通过”直接等同于“有没有稳定签名身份”，避免自签名 app 因为宿主文件系统元数据被误标成 `unsigned`。
- 重新检查当前手动分发产物体积：当前 `0.1.3` universal app 约 `11MB`，zip 约 `4.5MB`，DMG 约 `4.9MB`；其中主二进制约 `8.4MB`，内置默认音频 [PS5 Game Start.flac](../../MeetingCountdownApp/Resources/PS5%20Game%20Start.flac) 约 `1.6MB`，`Assets.car` 约 `944KB`。

### 6. 关键决策

- 菜单栏图标问题优先通过“回到系统模板渲染”解决，而不是继续手动指定黑色、白色或品牌色；因为副屏菜单栏灰态、浅深色切换和未来系统外观变化都应该优先交给 AppKit 自己处理。
- 现阶段不把“universal binary 导致体积略大”视为阻塞项。当前包体并没有达到异常体积，主增量确实主要来自双架构二进制，但总量仍处在可接受范围。

### 7. 遇到的问题

- 当前维护机上的 app bundle 会反复出现 `com.apple.FinderInfo`、`com.apple.provenance` 这类扩展属性；其中 `FinderInfo` 会让 `codesign --verify --deep --strict` 报 `resource fork, Finder information, or similar detritus not allowed`，看起来像“签名坏了”，但根因其实是文件系统元数据污染。
- 这次也确认：不能把 DMG / zip 的最终文件大小和“用户口头看到的 40MB”直接等同；仓库当前最新导出产物远小于这个数字，更可能是用户看到了解压后的 `.app`、旧版本产物、Finder 的“磁盘占用”口径，或包含额外调试产物的目录体积。
