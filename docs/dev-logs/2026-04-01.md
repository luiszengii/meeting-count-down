- 尝试给 `MenuBarExtra` label 做 `TimelineView` 动画时，发现会把测试宿主拖进高频重绘；最终回退到“提醒命中时静态切换菜单栏标签”，并补充 [docs/pitfalls/swiftui-menubarextra-timelineview-overdraw.md](../pitfalls/swiftui-menubarextra-timelineview-overdraw.md) 记录这条 SwiftUI 坑。
- 菜单栏基础图标不再回退成像设置入口的 `slider.horizontal.3`，而是统一改成日历 / 倒计时语义；提醒命中态则进一步收口为更醒目的粗体标签与告警图标。
- 由于通知中心横幅需要签名与系统授权配合，而当前产品决定保持“无需付费签名也可稳定使用”的本地闭环，后续又把这条能力从设置页、运行时和工程里整体移除，提醒链路重新收敛为“音效 + 菜单栏高优先级状态”。

### 2. 关键决策

- Phase 4 先只完成本地提醒引擎、菜单栏状态联动和默认音效，不在这一阶段引入系统通知和完整偏好设置 UI。
- 默认提醒音效改为 app bundle 内置的 `PS5 Game Start.flac`，程序生成短音只保留为兜底路径。
- 设置窗口只在用户显式点击“打开设置”时才允许前置；通知权限只在用户显式操作时申请，避免后台菜单栏 app 在提醒流程里触发系统权限问题。
- 把“系统通知横幅”降级为可选反馈，而不是核心提醒链路；核心提醒始终保留“音效 + 菜单栏状态变化”。
- 在评估 Developer ID 成本和本地 unsigned / ad-hoc build 的不稳定性后，最终把“通知中心横幅”完全移出当前产品范围，不再作为 V1/V1.1 方向继续保留。
- 提醒链路仍严格沿用 `CalDAV -> macOS Calendar -> EventKit -> SourceCoordinator -> ReminderEngine`，不回退到任何多接入模式。

### 3. 遇到的问题

- `SourceCoordinator` 在一次刷新过程中会多次发布状态，导致提醒引擎如果每次都先 `cancel` 再重建，会出现重复“Reminder scheduled”日志。当前已在 `ReminderEngine` 内通过等价上下文复用规避这一问题。
- 当前本地 ad-hoc 菜单栏 build 在部分机器上会被 macOS 直接拒绝通知授权，表现为 `UNErrorDomain error 1`；这也是最终把通知中心横幅移出主线实现的重要原因之一。
- `MenuBarExtra` label 不适合直接挂 `TimelineView` 做持续动画，否则很容易把宿主 app 拖进高频重绘；详见 [docs/pitfalls/swiftui-menubarextra-timelineview-overdraw.md](../pitfalls/swiftui-menubarextra-timelineview-overdraw.md)。
- 通知横幅这一支虽然已经验证过技术可行性，但它要求额外签名 / 系统授权前提，与当前“本地闭环、低门槛分发”的产品约束冲突，因此最终没有保留到主线实现中。
