# 2026-04-08

## 当天开发概括

参考 Andrej Karpathy 的 `llm-wiki` 思路，对当前仓库文档系统做了一次“内容导向入口”补强：保留现有相互链接和按天日志，但额外补上一份全局内容索引，降低文档查找成本。

## 详细内容

### 1. 完成事项

- 新增 [docs/index.md](../index.md)，作为“内容导向”的全局索引，集中列出当前项目计划内的正式文档，包括根级入口文档、`docs/` 主系统和模块级 `AGENTS.md`。
- 更新 [AGENTS.md](../../AGENTS.md)，把 `docs/index.md` 纳入正式文档结构，并明确区分“内容导向索引”和“时间导向日志”两种入口职责。
- 更新 [docs/README.md](../README.md)，把它从“半高层导航、半全量目录”的混合角色收敛成高层总览，并把完整清单入口明确指向 [docs/index.md](../index.md)。
- 更新 [docs/dev-logs/README.md](./README.md)，补齐此前遗漏的日志列表，避免子索引长期落后于真实文档集合。
- 修正 [docs/README.md](../README.md) 中指向单一天日志文件的陈旧入口，把它改成稳定的 [开发日志索引](./README.md)，避免“高层导航页自己先过期”。
- 新增 [本地自签名 Code Signing 身份用于手动分发](../pitfalls/local-self-signed-code-signing-identity-for-manual-distribution.md)，把此前散落在聊天里的维护者操作知识沉淀进正式 pitfall，并从 [manual-installation.md](../manual-installation.md) 和索引页接入。
- 更新 [AGENTS.md](../../AGENTS.md) 和 [docs/README.md](../README.md) 的使用规则，把“文档健康检查”正式写成仓库级治理要求，而不是只停留在一次临时排查里。
- 根据后续澄清，把 `.planning/` 和 `UI-REVIEW.md` 这类 GSD / review skill 生成的旁路工件从 [docs/index.md](../index.md) 的正式目录里剥离，避免把工具产物误当成项目计划内文档架构。
- 新增 [Tooling Artifacts](../tooling-artifacts.md)，把“正式文档 vs 工具产物”的边界单独写成一页，避免这个规则只分散在索引描述和聊天里。

### 2. 关键决策

- 不直接照搬 Karpathy 那套 `index.md + log.md + schema` 文件布局，而是做最小迁移：继续保留项目现有的 [AGENTS.md](../../AGENTS.md) 作为规则层、[docs/dev-logs/](./README.md) 作为时间日志层，再补一个 [docs/index.md](../index.md) 承担内容目录层。
- [docs/README.md](../README.md) 继续保留，但职责调整为“面向人类的高层导航页”，不再强行承担“列出全部文档”的维护负担；全量目录交给 [docs/index.md](../index.md)。
- 借鉴 Karpathy 提到的 `lint` 思路，但不额外引入单独命令文档；先把“定期做文档健康检查”固化为治理规则，后续再决定是否需要脚本化。
- 对工具产物采取“单独旁路层”而不是“彻底排除”：允许它们留在仓库里，但通过 [Tooling Artifacts](../tooling-artifacts.md) 明确声明它们不属于正式文档层。

### 3. 遇到的问题

- 当前文档系统虽然已经具备很强的相对链接网络，但入口分层不够清晰：高层总览、按天日志、专题索引和全量目录之间的职责边界此前没有被正式写清楚。
- 现有索引页存在轻微漂移，例如 [docs/dev-logs/README.md](./README.md) 长时间没有覆盖最新日志；这类问题说明项目需要一个更明确的“索引页也算正式文档，需要同步维护”的规则。
- 另外一个暴露出来的问题是：一些已经影响维护者真实操作的知识，仍然只存在于聊天记录里，例如“本地自签名代码签名 identity 从哪里来、怎么验证可用”。这类内容如果不落成正式文档，下一次仍然会重复口头排查。

### 4. 后续动作

- 后续新增、移动、归档任何持久 Markdown 文档时，同步更新 [docs/index.md](../index.md)，避免再次出现“文档存在，但没有统一入口”的情况。
- 继续做一轮文档健康检查，评估是否需要为 UI 审计、调试会话复盘或维护者操作指南增加更稳定的归档位置。
- 后续如果再出现“用户问题已经解决，但答案仍主要留在聊天里”的情况，优先考虑把它沉淀为 `pitfalls/`、维护者说明或 ADR，而不是只补一句开发日志。
