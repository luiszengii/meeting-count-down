### 2. 关键决策

- **本地 build 通过 ≠ ship 信号**：以后必须看 CI 才算数。本地 Swift 工具链版本可能比 CI 新，新版本对严格并发的某些 case 更宽松，会掩盖在 CI 上一定爆的问题。这条规则也补进 [AGENTS.md](../../AGENTS.md) 的 AI 代理执行约束之后单独评估是否需要硬性约束。
- **SwiftLint 阈值调整原则**：调宽默认值是为了把"真气味阈值"对齐项目的实际维护边界，不是为了消 warning 数量。对剩下的 2 条 `file_length`（`OverviewPage` 904 / `CalendarPage` 893）刻意不调阈值——它们就是该拆的 page，让 lint 继续提醒。
- **snapshot 测试是作者本地的回归保护，不是 CI 的视觉守护**：CI 跑 snapshot 是 anti-pattern；像素级 PNG 对比对字体/HiDPI/AppKit 版本敏感，CI runner 与作者本地必然有差异。snapshot 留在本地是有价值的（写代码时 catch UI 退化），CI 跳过是务实选择。详见 [snapshot ADR](../adrs/2026-04-23-snapshot-testing-framework.md) 的 Followup 段。
- **ReminderEngine deinit 使用普通 `deinit` 而非 `nonisolated deinit`**：后者是 Swift 6.1 实验性语法。普通 deinit 在 `@MainActor` 类型上默认就是 nonisolated，不需要任何特殊标注；只要 deinit 内访问的字段是 `nonisolated(unsafe)`，就是安全的。

### 3. 遇到的问题

- 最早从用户反馈"为什么 CI 全部 failed"才发现问题——之前一连串 commit (W6-W7-lint cleanup-...) 我都假设本地 build 过 = CI 也过，根本没检查 CI 状态。从 `831e6e8` 起 CI 已经红了 5 个 commit，期间用户自己的 `v0.1.5` 也是红的。详见 [swift-version-divergence-local-vs-ci](../pitfalls/swift-version-divergence-local-vs-ci.md)。
- 修完 Swift 6.0 严格并发后 CI 又因为 13 个 snapshot 测试全 fail 第二轮变红。第一反应是 baseline 错了，仔细看 diff 才意识到是跨环境像素差，本质上 snapshot 测试不该跑在 CI。详见 [snapshot-pixel-diff-cross-environment](../pitfalls/snapshot-pixel-diff-cross-environment.md)。
- 一开始用 BSD `sed -E 's/\bL(/localized(/g'` 做 5 文件 `L → localized` 重命名，跑完没报错也没生效——macOS 的 sed 不支持 `\b` 词边界（GNU sed 才支持）。grep 验证后才发现 sed 没改任何东西，换 `perl -i -pe` 才真完成。教训：批量改完一定 grep 验证，不能只看 sed 退出码。

### 6. 关键决策（hook 扩展部分）

- **检查顺序：AGENTS.md 在前，SwiftLint 在后**。AGENTS.md 检查只跑 `git diff --cached --name-only` 的列表对比，毫秒级；SwiftLint 要起 swiftlint 进程跑分析，秒级。把高频 + 轻量的检查前置，让常见违规立即暴露，节省后续 SwiftLint 调试循环。
- **CI 暂不跑 AGENTS.md 检查**。本地 hook 是第一道关；CI 的核心职责是保护"代码能编译能跑测试"。如果未来发现有人绕过 hook 推上来的 PR 仍在违反索引规则，再加 CI 步骤。
- **保留 `git commit --no-verify` 作为 escape hatch**，但在违规消息里明确写出"同时跳过 SwiftLint"的副作用，避免被滥用。
- **不试图自动修复**。`pre-commit-agents-check.sh` 只检测 + 报告 + 阻断，不去自动加 README/index 条目——因为 entry 描述需要人写（不能机械生成"这个文件干啥用"的一句话简介）。
