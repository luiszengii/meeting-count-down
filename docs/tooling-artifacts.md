# Tooling Artifacts

这份文档用来说明“正式项目文档”和“工具产物”之间的边界，避免把两类材料混在一起维护。

## 什么算正式文档

正式文档指的是：你明确认可、需要长期维护、会影响项目理解和交接的文档。当前正式文档体系包括：

- 根级 [README.md](../README.md) 和 [AGENTS.md](../AGENTS.md)
- `docs/` 下的正式项目文档
- 代码目录中的模块级 `AGENTS.md`

这些内容会进入 [Global Index](./index.md)，也属于文档健康检查的范围。

## 什么算工具产物

工具产物指的是：由 GSD / review skill / planning 流程自动生成，用来辅助某次分析、审计或调试的材料。它们可以保留在仓库里，但默认不属于正式项目文档体系。

当前仓库里的典型例子包括：

- `.planning/` 下的规划、调试和映射工件
- 根目录的 `UI-REVIEW.md`

这些内容默认：

- 不进入 [Global Index](./index.md) 的正式目录
- 不参与“正式文档是否完整”的判断
- 不要求按 `docs/` 模板持续维护

## 什么时候可以晋升

如果某份工具产物里的结论后来被你认定为长期有效、值得交接，就不要直接把原文件当成正式文档，而是把有效内容提炼后迁入 `docs/`：

- 调试结论优先进入 `docs/pitfalls/`
- 长期决策优先进入 `docs/adrs/`
- 操作说明优先进入已有维护者文档，例如 [manual-installation.md](./manual-installation.md)

原则是：工具产物可以提供原始素材，但正式文档要按项目语境重写。
