# 2026-03-30 建立项目级文档治理系统

- 状态：Accepted
- 日期：2026-03-30
- 关联日志：[2026-03-30 开发日志](../dev-logs/2026-03-30.md)

## 背景

项目尚未正式进入 Swift/macOS 客户端开发，但已经确认需要借开发过程系统性学习 Swift 生态。仅靠聊天记录无法长期承载知识，且新 session 无法天然继承口头约定。如果不在项目启动前就建立文档治理系统，后续代码一多，模块边界、架构决策、踩坑经验和上下文会迅速碎片化。

## 决策

项目从现在开始采用“仓库级规则 + 目录级索引 + docs 文档网络 + 模块级中文代码注释”的文档治理方式：

1. 根级 `AGENTS.md` 作为仓库级持久规则入口。
2. 新增代码目录时，先创建目录级 `AGENTS.md`，再写核心代码。
3. 核心代码文件必须包含中文模块级详细注释；后续细化为函数级注释默认必写，整体注释密度目标约为每 `8` 到 `10` 行有效代码至少有 `1` 行高信息量注释。
4. `docs/` 固定维护 `dev-logs/`、`pitfalls/`、`adrs/`、`templates/`。
5. 开发日志按天维护；技术和产品形态变动用 ADR；反复推敲的问题用 Pitfall。
6. 文档之间统一使用相对路径 Markdown 链接。

## 备选方案

### 方案 A：只保留根级 AGENTS 和 README

优点是简单；缺点是知识会过度集中在两个文件里，随着开发推进很快失控，也不利于记录演进过程。

### 方案 B：只维护 docs，不做目录级 AGENTS

优点是文件少；缺点是代码树本身缺少局部索引，后续进入某个模块目录时仍然需要跳回仓库根找上下文，不利于树状传递。

### 方案 C：大量逐行代码注释，不强调 docs

优点是代码旁边就能看到解释；缺点是注释会变成噪音，且无法承载决策历史、排查过程和跨文件知识。

## 影响

- 开发前期会增加一些文档维护成本。
- 后续每次新 session 进入仓库时，都可以通过根级 `AGENTS.md` 快速恢复规则。
- 模块边界、架构决策、排障经验、学习资料将逐步沉淀为长期资产，而不是散落在聊天记录中。

## 后续动作

1. 正式开发代码前，保持 `docs/` 模板与根级 `AGENTS.md` 一致。
2. 一旦创建实际代码目录，先补目录级 `AGENTS.md`。
3. 首个 Swift 模块落地时，严格执行中文模块级详细注释规则。
