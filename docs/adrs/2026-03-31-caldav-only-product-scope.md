# 2026-03-31 CalDAV-only 产品范围收敛

- 状态：Accepted
- 日期：2026-03-31
- 关联日志：[2026-03-31 开发日志](../dev-logs/2026-03-31.md)
- 补充说明：这条决策会覆盖此前关于多路接入、`BYO Feishu App`、`Offline Import`、`lark-cli` 的产品级规划，但不等于代码已经完成清理；代码收敛需要后续实现阶段继续推进。

## 背景

项目早期曾经讨论过“四路接入”方案：`CalDAV`、`BYO Feishu App`、`Offline Import`、`lark-cli`。这种设想在探索阶段有价值，因为它帮助梳理了不同权限条件下可能的接入路径。

但进入真正开发前，问题已经很明确：

1. 真正最像“普通用户可走通路径”的只有 `CalDAV -> macOS Calendar`。
2. `BYO Feishu App` 对用户的开放平台能力、企业权限和 OAuth 配置要求太高，会显著抬高使用门槛。
3. `Offline Import` 虽然能兜底，但会把产品从“自动提醒”拉回“手动维护文件”。
4. `lark-cli` 会受审批、安装和环境差异影响，不适合作为最终面向普通用户的运行时主链路。
5. 如果继续同时维护多条路线，文档、代码骨架、状态模型和测试计划都会持续发散，真正可交付的一条主路径反而更慢。

## 决策

从 `2026-03-31` 起，项目产品范围正式收敛为 `CalDAV-only`：

1. 只保留 `CalDAV -> macOS Calendar -> EventKit` 这一条飞书会议接入路径。
2. 不再继续规划或扩展 `BYO Feishu App`、`Offline Import`、`lark-cli` 作为会议接入入口。
3. 根级 `AGENTS.md`、`README.md`、后续实现计划、测试计划都要围绕 CalDAV 单一路线重写。
4. 历史上遗留的多路接入命名、模块和状态骨架应视为待收敛历史遗留，而不是未来能力清单。

## 备选方案

### 方案 A：继续保留多路接入，只是把 CalDAV 标成推荐路径

没有采用。原因是这会继续保留过多历史包袱，导致文档和代码长期处于“理论上支持很多，实际上只做一条”的混乱状态。

### 方案 B：只保留 CalDAV，同时保留离线导入作为兜底

没有采用。原因是当前最重要的是先把自动提醒的主路径做稳定，而不是继续为低频兜底场景维护第二套数据入口。

## 影响

- 产品定位从“多路接入 meeting reminder app”收敛成“CalDAV 同步后的 macOS 系统日历提醒 app”。
- 首次启动流程不再是接入方式选择器，而是 CalDAV 配置说明、系统日历权限检查和目标日历选择。
- 架构重点从 OAuth、CLI、导入模块转移到 `EventKit`、系统日历变化监听、同步新鲜度提示和提醒调度。
- 仓库里如果还存在多路接入命名，需要后续逐步删除或收敛，而不是继续往上叠功能。

## 后续动作

1. 更新根级 `AGENTS.md` 与 `README.md`，移除多路接入叙述。
2. 后续代码实现优先收敛 `ConnectionMode`、多模式文案和多源状态机。
3. 后续测试与设置页设计只围绕 CalDAV 路径展开。
