# 2026-04-02 Phase 6 先转为无会员手动分发

- 状态：Accepted
- 日期：2026-04-02
- 关联日志：[2026-04-02 开发日志](../dev-logs/2026-04-02.md)

## 背景

项目原本把 `Phase 6` 直接定义成“`Developer ID` 签名、Apple notarization、DMG 和 Sparkle 自动更新”。这条路线适合正式对外发布，但它有一个很现实的前提：需要加入 Apple Developer Program。

当前阶段的问题不是技术上完全做不到，而是这条分发链路会立刻引入持续成本和额外配置负担：

1. 没有付费开发者会员，就无法获得正式的 `Developer ID` 签名和 notarization 能力。
2. 如果继续把 `Phase 6` 定义成“先完成正式签名发布”，当前迭代会直接被预算问题卡住。
3. 项目现阶段更需要的是把“别人能不能实际装起来、按文档接入 CalDAV、稳定收到提醒”先验证出来。
4. 对当前这类菜单栏本地工具来说，短期内“可手动安装的小范围测试版”比“尚未开始的正式分发方案”更有推进价值。

## 决策

从 `2026-04-02` 起，项目把 `Phase 6` 调整为“无开发者会员前提下的手动分发准备”：

1. 当前阶段优先完成 `Release` 构建、`.app` / `.zip` 导出和首次打开放行说明。
2. 对外接入文档继续推进，但面向的是“愿意手动放行 app 的测试用户”，而不是零门槛公众下载。
3. `Developer ID` 签名、Apple notarization、DMG 和 Sparkle 自动更新不再视为当前阶段阻塞项，延后到未来预算允许时再恢复推进。
4. 文档中所有“当前阶段 / 当前里程碑 / 平台与分发”描述都要同步改成这条更现实的分发路线。

## 备选方案

### 方案 A：维持原定义，坚持先做正式签名发布

没有采用。原因是这会让当前阶段长期停在“目标正确但无法开始”的状态，实际推进效率接近于零。

### 方案 B：完全放弃任何分发，只保留本机 Xcode 运行

没有采用。原因是这样无法验证普通用户视角下的安装步骤、CalDAV 配置理解成本和手动分发时的真实门槛。

## 影响

- `Phase 6` 的目标从“正式分发”收缩成“手动分发与安装文档”。
- `Phase 7` 的测试方式也从默认公开测试，变成更现实的小范围手动安装测试。
- 当前发布链路不会接入 Sparkle、appcast、notarization 等正式分发基础设施。
- 如果未来预算允许恢复付费会员，仍然可以在现有文档和构建基础上继续补正式签名与自动更新，而不是推翻现有主线。

## 后续动作

1. 更新根级 `AGENTS.md`、`README.md` 和 `docs/README.md`，统一当前阶段描述。
2. 补一份面向测试用户的手动安装与首次打开放行说明。
3. 增加本地 `Release` 构建和导出步骤，确保可以稳定产出手动分发包。
4. 后续如果预算允许，再单独新增 ADR 恢复 `Developer ID` / notarization / Sparkle 正式分发路线。
